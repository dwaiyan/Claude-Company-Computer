# ADR-0001: 微服务架构选型

**日期:** 2026-06-09  
**状态:** 已采纳  
**决策者:** 技术团队

## 背景

技术能力提升平台需要一个既可快速上线、又能作为 .NET 微服务教学示范的架构。团队主技术栈为 .NET 6 + Vue 3 + PostgreSQL/MySQL + Redis + RabbitMQ。

## 决策

采用 **微服务原生架构**，按业务域拆分为 6 个独立服务，通过 YARP 网关统一入口。

### 服务边界

| 服务 | 端口 | 数据库 | 业务职责 |
|------|------|--------|----------|
| Identity | 5001 | PostgreSQL | 用户、角色、权限、JWT 认证 |
| Content | 5002 | PostgreSQL | 技术树/节点、学习资源、面试题 |
| Learning | 5003 | MySQL | 学习路线、技能雷达、打卡、评测 |
| Collaboration | 5004 | PostgreSQL | 笔记、讨论、内容贡献审核 |
| Notification | 5005 | Redis | 站内通知、事件消费 |
| CodeRunner | 5006 | (无状态) | .NET 代码在线编译运行 |

### 关键决策

1. **YARP 作为 API 网关** — 与 .NET 生态一致，零额外运维成本
2. **PostgreSQL vs MySQL 混用** — Learning 服务用 MySQL 是为了教学示范异构数据库场景
3. **同步 HTTP + 异步 RabbitMQ** — 查询走 REST，通知走消息队列
4. **共享包 SkillPlatform.Common** — 避免代码重复，统一错误处理/JWT/日志

## 后果

- **优势：** 各服务独立部署/扩展/测试；天然模块化；与团队真实技术方向一致
- **代价：** 本地开发需启动多个服务 + PostgreSQL + MySQL + Redis + RabbitMQ
- **风险：** 服务间调用延迟、分布式事务复杂性（P1-P4 阶段尚未遇到，预留 Polly 韧性策略）

## 备选方案

- **模块化单体（被否决）：** 对内部工具来说足够，但无法作为微服务教学的活案例
- **单体外加网关（被否决）：** 无法展示真正的微服务治理
